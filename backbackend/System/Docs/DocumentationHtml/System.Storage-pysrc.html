  1  ''' 
  2  =========== 
  3  Routines for storing and retrieving information about files and python objects.  
  4  =========== 
  5   
  6  In the Data Environment, which is devoted to understanding and managing dependency links,  
  7  routines throughout the system often require stored information about files, directories,  
  8  functions, and data structures.   This information may include: 
  9                   
 10  1) Information that can't directly be determined by analyzing the file or directory:   
 11  for example, the file's last modification-time or create-time.    This information is  
 12  often necessary to determine when an object has changed, so as to know when to  
 13  trigger the re-running of a process that depends on the object.   This is the case 
 14  with the function PropagateThroughLinkGraphWithTimes, in the LinkManagement  
 15  module, which needs to compare link target's mod times to link source's mod  
 16  times to know when to re-run a data-creation script.  
 17   
 18  or,              
 19   
 20  2) Information that is that _is_ possible to glean upon complex introspection but hard  
 21  to obtain otherwise: for example, the list of names of python functions that might be called  
 22  by another function during its run.  This information is useful to analyze an object to  
 23  determine what its dependencies are in the first place.   This is the case with the function  
 24  ComputeLinksFromOperations, which determines the data "depends_on" and "creates",  
 25  and functional "uses",  from a python function's code.   
 26                   
 27  This module provides a unified interface to obtain such information so that  it can be  
 28  called up wherever needed.   The key realizaion is that providing unified access to the 
 29  relevant information about the objects boils down to having a unified method for  
 30  _storing_ and retrieving "stylized versions" of those objects,  versions that contain  
 31  somewhat detailed information about the objects' parts, as well as information about 
 32  when those parts were last modified.    
 33   
 34  This is because: 
 35   
 36  1) if you need to get information about when an object has changed, you'll need to have a  
 37  stored version that can be compared to the current version to detect chages, together  
 38  with timing information about each of those stored parts so that if the part _hasn't_  
 39  changed you can tell what the last actual mod-time of the part was.  
 40   
 41  and      
 42   
 43  2) to provide easy access to complex introspected information, it makes sense to  
 44  compute that introspection only once per object per modification, and store the 
 45  results of that introspection in a standardized format in a single place.  
 46                   
 47  The  "data model" behind our approach to doing this enables us to be as lazy as possible,  
 48  leaving as much to operating system as we can.  The basic idea is that every object  
 49  stored on disk in the Data Environment file system is one of two things: 
 50          1) A directory or a file inside a directory  -- whose properties are "at the operating system level"  
 51                  and which don't need to be made "live" to access 
 52  or 
 53          2) an implied object __inside__ a file -- whose properties are "more specific than 
 54          the operating system level" and which require some form of "being live" to access 
 55                   
 56  For instance, a python package consists of a directory containing python .py files, and 
 57  in turn each py file is a module containing python objects.  Or a relational database,  
 58  which is at one level a filesystem, but which at a lower level consists of records.  
 59   
 60  Now, for information about things at the files and directories levels, we are able to  
 61  rely mostly on things provided by the operating system: it obviously already stores  
 62  the files themselves, and provides access to file and directory modification  
 63  information, through things like the 'stat' and 'diff' utilities.     However, for finer  
 64  detail, we need to supplement the operating system.   The basic strategy is: 
 65          -- for each of several "Special Supported File Types", store information about the  
 66          more detailed parts of the supported file type 
 67  and then 
 68          -- provide an _extension_ of the stat and diff utilities that allows queries to specify  
 69          both a file path, as well as a more detailed  part-name. 
 70   
 71  For instance, the standard python implementation of the path mod-time function is  
 72  'getmtime' function in the the os.path module.  The function os.path.getmtime takes 
 73  as argument a single path, and returns the time of last modification of that path.    
 74  Here, we extend that function to the FindMtime function which takes both a pathname  
 75  argument, as well as an "object-name" argument, and which returns the mod-time of  
 76  that specific object.  For the moment, we only have one "Special Supported File Type":   
 77  python modules ; so the "objectname" argument boils down to a "function-name"  
 78  argument that allows one to find the mod-time for a given function in a module.    
 79  All other files are treated at the atomic level.  (In the future, other kinds of file parts 
 80  could be stored, e.g. records in a database by integrating database query routines .... etc...)
