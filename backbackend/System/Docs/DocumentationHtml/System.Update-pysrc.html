  1  ''' 
  2  Functions for automatic updating of data in the Data Environment by calling  
  3  scripts indicated by data dependency links.  
  4           
  5  The basic flow of information is that the functions in this module are  
  6  fed by data produced by the functions from the module  
  7  System.LinkManagement module.  Specifically, LinkManagement functions 
  8  (like GetLinksBelow) generate sequences of data dependency links  
  9  representing progations up or downward in the directed acyclic graph 
 10  formed by the set of all Data Dependency links.  These functions  
 11  can be passed parameters that have them qualify the links they return  
 12  by timestamps to show only those links that actually _need_ updating,  
 13  or be unqualified and actually show the full down- or upstream propagations.  
 14   
 15  The data dependency link sequences come in the following form: 
 16   
 17          [L1, L2, L3 ... , LN] 
 18   
 19  where each LI is (essentially) a numpy record array of dependency  
 20  links meant to be called at stage i of an update -- because things at  
 21  stage i-1 are input dependencies to things at stage i, which are in turn  
 22  inputs to things at stage i + 1.   Within the LI, the records links  
 23  represented (essentially) as: 
 24   
 25          (Source,Target,Script) 
 26   
 27  where "Script" is the procedure that needs to be called to produce 
 28  Target from Source.  
 29   
 30  Once the sequences of links are returned to functions here, the  
 31  UpdateLInks function calls the scripts indicated by these links, 
 32  in the rounds indicated by the sequence.  
 33   
 34  There are two basic approaches to automatic updating that are  
 35  supported by the routines provided here. 
 36   
 37  1) "Downstream Updating," in which the user asks the system to detect  
 38  changes have been made, either to data or scripts, since an update was  
 39  last run, and then propagate the results of these changes downstream  
 40  through the data dependency graph.  This functionality is implemented 
 41  by the "FullUpdate" function.  
 42           
 43  2) "Upstream Updating," in which the user specifies a target or set of 
 44  targets that he wishes to remake; then the system determines which  
 45  upstream elements have been modified since the last creation of the  
 46  target, and runs those that need updating in the proper order.  This  
 47  functionality -- which is similar in spirit to the traditional "makefile"  
 48  commands available for many code-compilation schemes -- is 
 49  implemented by the "MakeUpdated" function.  
 50   
 51  The "FindOutWhatWillUpdate" function displays what scripts will be  
 52  called, given various options to the updater, without actually calling 
 53  any updates.  
 54   
 55  The user can blend the two updating styles along a seemless 
 56  continuum.  The is done is via Automatic Updates filtering.  The  
 57  user specifies in a configuration file (whose path is set at the  
 58  AutomaticUpdatesPath environment variable) those scripts or  
 59  script-name patterns that should or should not be updated upon 
 60  downstream propation.  Then, when calling a MakeUpdated command 
 61  to a set of targets, Upstream changes are included, as well as  
 62  downstream propagations from those upstream changes that are  
 63  allowed by the Automatic Updates settings.    
 64   
 65  The advantage of Downstream Updating is that it does not require  
 66  the user to think about the implications of the changes he's making:  
 67  he merely makes changes to data and scripts, and then calls FullUpdate, 
 68  regardless of what the changes were; he does not have to remember  
 69  which files the changes are bound to affect and then make sure  
 70  those are updated.  On the other hand, this style of updating means  
 71  that the user ends up having to modify the "Automatic Updates"  
 72  settings occasionally to prevent unwanted updates from occuring,  
 73  as well to enable wanted updates to actual happen.  It also means  
 74  that he often will want to call "FindOutWhatWillUpdate" before  
 75  "FullUpdate" to make see ahead of time to see which unwated 
 76  automatic updates (if any) he may have to disable.    Downstream 
 77  Update "purists" will typically want to set the "Automatic Updates" 
 78  file to enable updating of most of what they're working on at any 
 79  given period, and then only call the two commands "FindOutWhatWillUpdate" 
 80  and "FullUpdate" repeatedly.  
 81   
 82  The Upstream style has the advantage of being more targeted -- and  
 83  requiring no interaction (if not wanted) with the automatic update  
 84  settings.   The Upstream Update "purists" -- those for example who  
 85  may be familiar with the idea of Makefiles and want to use that style 
 86  in the data analysis work -- will typically want to set the Automatic  
 87  Updates setting to contain little or nothing, and then call  
 88  MakeUpdated(Target) for whatever end-point target they ultimately  
 89  want to keep updated.  
 90   
 91  In either case, the user can _force_ updating by setting the Force options 
 92  to True (this is like the "make clean" option in make systems); or have 
 93  the system update just those things that the system thinks really need  
 94  updating.
